# System Architecture Design

## 1. Overview
The Kanpaku System is an autonomous, hierarchical multi-agent orchestration framework designed for project management and execution using Consistency-Driven Development (CoDD) principles. The system implements a strict division of labor inspired by the Heian-era imperial court, ensuring clear accountability and task isolation. The primary goal is to provide the Mikado (User) with a single interface to manage complex, multi-step projects where tasks are automatically decomposed, assigned to specialized agents, executed within secure sandboxes, and reviewed against rigorous quality thresholds.

The system is deployed on an infrastructure stack consisting of an NVIDIA RTX 2070 SUPER (8GB VRAM), 64GB DDR4 RAM, and an Intel Core i5-13600KF. It utilizes the **infra:llm-stack** via Ollama and llama.cpp to run specialized models, including `gemma4-26B-A4B` for high-level reasoning and `Bonsai-8B` for execution. System state is managed via Redis for real-time operations, Chroma DB for skill persistence, and YAML for historical auditability.

## 2. Architecture

### 2.1 Agent Hierarchy and Model Mapping
The system complies with the **infra:llm-stack** constraint by mapping specific models to roles based on Bloom’s Taxonomy and computational requirements:

*   **Mikado (User):** The external trigger and monitor. Interfaces exclusively through the **module:kanpaku** dashboard.
*   **Kanpaku (Goal Setter):** Acts as the sole interface for the Mikado (**module:kanpaku**). It uses `gemma4-26B-A4B` (Ollama) to process high-level instructions, generate `ORDER_CREATED` events, and determine if the current project requires the creation of new skills.
*   **Jiju (Manager):** Managed by **module:jiju**, this layer handles task decomposition and agent assignment. It uses `gemma4-E2B` (Ollama) and runs in multiple processes to monitor heartbeats and update task statuses. It is responsible for calling `codd review` and `codd validate`.
*   **Toneri (Executor):** Handles Bloom Level L1-L3 tasks (Remember, Understand, Apply). It utilizes `Bonsai-8B`, `DeepSeek 6.7B`, or `gemma4-E2B` (llama.cpp/Ollama). These agents have file-system access within the sandbox.
*   **Onmyoji (Analyzer):** Handles Bloom Level L4-L6 tasks (Analyze, Evaluate, Create). It utilizes `gemma4-26B-A4B` to extract structural patterns and perform architectural reviews. Onmyoji are restricted from direct file execution.

### 2.2 Task Lifecycle and State Management
Tasks transition through a rigorous state machine maintained in Redis and persisted to YAML.
1.  **ORDER_CREATED:** Generated by Kanpaku from Mikado input.
2.  **TASK_CREATED:** Jiju decomposes orders into discrete tasks with a priority (1-100) and Bloom level.
3.  **ASSIGNED/DOING:** Toneri or Onmyoji pick up tasks from their respective `inbox:{agent_id}` streams.
4.  **REVIEWING:** Jiju evaluates the result. A `review.score` >= 80 is mandatory for completion (Ref: `review:score-threshold`).
5.  **COMPLETED/FAILED:** Successful tasks are persisted to `/history/tasks/T{id}.yaml`. Tasks failing 10 consecutive retries are moved to `FAILED`.

### 2.3 Data Structures and Persistence
*   **Redis (Live State):**
    *   `tasks:{task_id}`: Hash containing `status`, `assigned_agent`, `priority`, and `review_score`.
    *   `agents:{agent_id}`: Hash containing `status` (idle, thinking, working, error), `current_task_id`, and `last_heartbeat`.
    *   `lock:{filepath}`: Distributed lock with a 300s TTL to prevent concurrent writes to the same file.
*   **Chroma DB (Vector Store):**
    *   `skills` collection: Stores embedded JSON documents of successful patterns. Metadata includes `success_rate` and `usage_count`.
*   **File System:**
    *   `/project/`: The strictly enforced sandbox directory.
    *   `/history/tasks/`: Persistent YAML records of all completed/failed tasks.
    *   `/logs/state.log`: Append-only log of system-wide state transitions.

### 2.4 Operational Constraints and Security
To ensure reliability and compliance with the `agent:timeout-policy`, the following thresholds are hard-coded into the orchestration logic:
*   **Thinking Timeout:** 300 seconds. If LLM inference exceeds this, a `TASK_STALLED` event is triggered.
*   **Doing Timeout:** 120 seconds. If I/O or execution exceeds this, a `TASK_STALLED` event is triggered.
*   **Heartbeat Interval:** 30 seconds. Agents must update their `last_heartbeat` in Redis or be marked as `error`.
*   **Sandbox Isolation:** Every file operation path is checked against the `/project/` prefix. Any attempt to access `/etc/`, `~/.`, or other system paths results in an immediate `Forbidden` exception and task failure.
*   **Concurrency:** Write operations require the acquisition of a Redis `lock:{filepath}`. If the lock is held, the agent must wait or retry after a backoff.

### 2.5 Skill Evolution Workflow
The system improves its own performance through a post-task analysis cycle:
1.  Upon a `TASK_COMPLETED` status with score >= 80, Kanpaku triggers an `ANALYZE_CREATING` event.
2.  The Onmyoji agent analyzes the task execution logs and extracted output to identify a "Skill Pattern."
3.  The pattern is embedded and stored in Chroma DB.
4.  During the `TASK_CREATED` phase for future tasks, **module:jiju** retrieves the top 3 similar skills and injects them into the agent's prompt context to reduce error rates.

## 3. Open Questions
1.  **Mobile Interface:** While the Mikado manages the system via the Kanpaku interface, the specific implementation details for the Android management application (push notification triggers for `TASK_FAILED`) are currently undefined.
2.  **Fine-Tuning Trigger:** The exact threshold of "accumulated patterns" required to trigger LoRA fine-tuning for `Bonsai-8B` using historical data in `/history/tasks/` needs to be quantified.
3.  **VRAM Budgeting:** With multiple agents running in parallel across Ollama and llama.cpp, a dynamic VRAM allocator is needed to prevent OOM errors when both `gemma4-26B-A4B` and `Bonsai-8B` are active simultaneously.
4.  **Archival Logic:** The criteria for moving YAML history from `/history/tasks/` to secondary cold storage after the 30-day window is not yet implemented in the current task cleanup routine.
